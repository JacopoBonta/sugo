## General

The majority of the syntax rules of Go get enforced automatically by the `go-fmt` executable, which gets run by the code editor if the Go extension is installed.

### Go installation

**GOPATH**
After having installed Go following [the official instructions](https://go.dev/doc/install), you might need to check if you have the `GOPATH` environment variable in your default shell.

If not, you need to add it.
The location of this `GOPATH` needs to be a folder for Go to understand where to download dependency modules.
It is usually located into `$HOME/go`.

**Binaries folder**
When you install global modules (using the `go install` command) you need to reference the `$GOPATH/bin` folder into your existing `$PATH` environment variable, so the binaries could get located.

### Visual Studio Code - Go extension

You can download the Go extension at [this URL](https://marketplace.visualstudio.com/items?itemName=golang.Go).
It provides syntax highlighting and automatic code lint and code fix on file save.

### Bazel compilation

Even for new projects that don't need to become microservices runnable in Tilt, it's always preferrable to build with `bazel`.

#### How the bazel toolchain works with Go

The `bazel` toolchain automatically downloads the agreed version of Go (ideally the latest) and builds the executable on its own, but to do so it needs some other information provided by the `BUILD` files scattered in the whole codebase, written in `Starlark`, a `Python` dialect.

These `BUILD` files can (and must) be autogenerated by a tool called `gazelle`, which purpose is to provide the missing link between `Go` and `bazel` by creating all the required `Go` compilation instructions for `bazel`.

**Gazelle**
Gazelle should be launched **whenever we create a new `Go` file or project**, in order for it to get indexed for the next `bazel compilation`.

The command is `bazel run //:gazelle`.
It produces a `BUILD` file in each folder containing `.go` files, using two rules: `go_library` and `go_binary`.

These autogenerated rules usually don't need to be changed, apart from the `BUILD` file of the path containing the entrypoint of the application, which needs a custom `bazel` rule defined by us instead, which would result in having these two rules instead: `go_library` and `go_services` (for an example check `coordinator/services/gods3-message-handler/cmd/message-handler/BUILD` or `coordinator/services/iam/BUILD`).

Even though these files are autogenerated, running `gazelle` again won't delete the rules we provided on our own.

**Gazelle's "update repos"**
What about the imported libraries we declare in our `go.mod`? Do they need their `BUILD` files too?
Well, sort-of.
For this purpose there's a command called `gazelle-update-repos` which needs to be run **whenever we update a `go.mod` or a `go.mod.sum`**.

The command is `bazel run //:gazelle-update-repos`.
It produces an import instruction in the file `coordinator/third_party/go/repositories.bzl` for every external library we need on our applications: it's like a `go.work.sum`, but on the `bazel` side.

### Best practices when working with Go

First thing first, ensure you have the same version of `Go` as the one used in the bazel monorepo: you can find the version by looking in the `/WORKSPACE` file and looking for

```python
go_register_toolchains(version = "1.x")
```

then, follow these simple rules when working with Go code:

- when creating a new `Go` project, make sure you are declaring it with the version agreed on other projects (e.g. use `go 1.20` in your `go.mod` if the other services are on the `1.20`)
- **always** run `bazel run //:gazelle-update-repos` after a `go mod tidy` (they do the same thing on two different levels of abstraction)
- **always** run `go mod tidy` when you add a dependency, even when the dependency is another package in the monorepo
- **always** run `bazel run //:gazelle` after creating/renaming/deleting a `.go` file

To enforce these rules there is a piece of configuration you can add on your `tilt_config.json` that enables the automatic run of these command at every build of a Go microservice, **when ran inside Tilt**.

To use it, just copy-paste the following json code in your `coordinator/tilt_config.json`:

```json
"toolchain": {
    "go": {
        "auto_run_tidy": true,
        "auto_run_gazelle": true
    }
}
```

## Syntactic guidelines

You can follow the style guide rules defined at [this page](https://google.github.io/styleguide/go/index).
They represent a comprehensive list of guidelines to apply to your code for maintainability and consistency.

Some of these guidelines could be summarized as follows:

### Simplicity

Your Go code should be as simple to read and maintain as possible, so you should avoid complex abstractions.
You should avoid high cognitive load by assuming that the reader can memorize all of the preceding code.

The code should have comments that explain why, not what, a decision has been made.
The code has useful errors and useful test failures.

### Concision

You should use concise code utilizing common idioms.
For example, the following code is very common in error handling:

```go

// ✅ Correct

if err := doSomething(); err != nil {
    // handle ..
}

```

## Formatting

### Camel case

Go code uses `CamelCase` rather than underscores.
This applies also to conventions that we are used to in other languages, like when declaring constants: you should use `MaxLength`, not `MAX_LENGTH`.

### Naming

In Go names tend to be somewhat shorter than in many other languages, strongly taking into consideration the context they are declared in.

## Error handling

### Swallowing

Errors should **never** be swallowed or replaced by new ones without meaningful information.
It's also a best practice to wrap errors to identify the main cause of an issue in your code, but without redundant details:

```go

// ❌ Incorrect: error has been replaced by a new one
// What it is the main cause of it?

if err := doSomething(); err != nil {
    return errors.New("an error has happened")
}
```

```go

// ✅ Correct

if err := doSomething(); err != nil {
    return fmt.Errorf("error performing doSomething: %w", err)
}
```

```go

// ❌ Incorrect: we are wrapping the error, but details are redudant.
// Output: "could not open settings.txt: open settings.txt: no such file or directory"

if err := os.Open("settings.txt"); err != nil {
    return fmt.Errorf("could not open settings.txt: %w", err)
}
```

```go

// ✅ Correct

if err := os.Open("settings.txt"); err != nil {
    // This error message disambiguates the call from many other ones
    // that could happen in this block of code
    return fmt.Errorf("launch codes unavailable: %v", err)
}
```

## Idiomatic code

### Options pattern

You should embrace the usage of idiomatic code.
For example, you might need something resembling default values for parameters in a constructor function.

In typescript you would do this like as follows:

```typescript
public class Buildable
{
    public static Build(name: string, surname = 'nosurname') {
        // ...
    }
}
```

In Go there's no support for default parameter values, so you will need to use the [Options Pattern](https://levelup.gitconnected.com/options-pattern-in-golang-9a0384a9d8db):

```go
// server/server.go
package server

type Server struct {
    host string
    port int
}

func New(options ...func(*Server)) *Server {
    srv := &Server{}

    for _, o := range options {
        o(srv)
    }

    return srv
}

func WithHost(host string) func(*Server) {
    return func(s *Server) {
        s.host = host
    }
}

func WithPort(port int) func(*Server) {
    return func(s *Server) {
        s.port = port
    }
}
```

```go
// main.go
package main

func main() {
    srv := server.New(WithHost("localhost"), WithPort(9146))
}
```
